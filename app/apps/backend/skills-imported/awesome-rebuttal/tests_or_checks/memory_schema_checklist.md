# Memory Schema Checklist

- [ ] `.awesome-rebuttal/memory/project_memory.json` records skill_state, workspace, venue, paper source, LaTeX environment, template, versioning, canonical response format, language policy, open questions, and source index.
- [ ] `.awesome-rebuttal/memory/venue_rules.json` records status/source, canonical `response.mode`, platform, limits, response permissions, formatting, content permissions, anonymity, confirmation, language policy, unknowns, and raw rule excerpt when available. AI-found rules stay `ai_found_pending_confirmation` until user confirmation.
- [ ] `.awesome-rebuttal/memory/paper_memory.json` records paper identity, source status, narrative/storyline, contribution ledger, method memory, experiment matrix, result ledger, figure/table index, terminology, limitations/weak points, rebuttal relevance hooks, and claim-evidence map.
- [ ] `.awesome-rebuttal/memory/code_memory.json` records code availability, framework stack, architecture map, innovation implementation map with insertion points, config/hyperparameter ledger, entrypoints/commands, dataset/evaluation pipelines, result logs, claim-code map or provisional code-claim map, experiment feasibility, and reproducibility risks.
- [ ] `.awesome-rebuttal/memory/review_memory.json` records raw reviewer memory, stable labels plus anonymous reviewer IDs, section index, raw anchor map, neutral issue item index, common issue index, parser notes, and missing metadata without strategic severity labels.
- [ ] `.awesome-rebuttal/memory/review_analysis_memory.json` records reviewer semantic summaries, semantic concern cards, semantic common clusters, evidence availability, and coverage audit without priority ranking.
- [ ] `.awesome-rebuttal/memory/experiment_memory.json` records numbered `EXP-*` candidates from reviewer concerns, reviewer/concern links, feasibility, venue compatibility, risk, triage class, result status, response usage plan, and user decision state.
- [ ] `.awesome-rebuttal/memory/strategy_memory.json` records posture, priority-ranked concerns, experiment priority links, reviewer goal model, top focused problems, global response themes, per-reviewer strategy, experiment strategy mapping/dependencies, paper-edit handoffs, coverage plan, tone plan, strategy matrix, user strategy decision/questionnaire, response blueprint, and AC summary decision.
- [ ] `.awesome-rebuttal/memory/template_memory.json` records response mode, layout family, source priority, fallback/official status, active template or draft output directory, generated files, LaTeX/render status, and consistency checks.
- [ ] `.awesome-rebuttal/memory/response_memory.json` records canonical response mode, language policy, response units, reviewer/concern links, evidence anchors, numeric claim verification, experiment usage, compression decisions, coverage map, draft outputs, and safety precheck.
- [ ] `.awesome-rebuttal/memory/ac_summary_memory.json` records AC summary decision, venue-rule status, shared positive facts, low-score context, fact provenance, draft variants, user confirmation, and no-attack/no-ranking/rule/space risk checks.
- [ ] `.awesome-rebuttal/memory/safety_memory.json` records final gate statuses, claim audit, blocking fixes, and route-back plan.
- [ ] `.awesome-rebuttal/memory/versioning_memory.json` records selected mode, git/snapshot state, checkpoint history, safe blocked actions, and next checkpoint.
- [ ] `.awesome-rebuttal/memory/overleaf_sync_memory.json` records Overleaf/LeafLink sync decision, base URL, project reference type, target/local dir, environment manager/name, install/auth status, allowed operations, command plan/run log, files/PDF detected, and submission-facing contamination check when relevant.
- [ ] `.awesome-rebuttal/snapshots/snapshot_memory.json` records detailed latest snapshot, source freshness, memory status, review/situation/strategy/experiment/draft/safety digests, next actions, and newest-first history.
- [ ] `.awesome-rebuttal/snapshots/REBUTTAL_SNAPSHOT.md` or `PROJECT_SNAPSHOT.md` is generated from `snapshot_memory.json`, has exact timestamps, and matches the same latest snapshot ID/stage/status.
