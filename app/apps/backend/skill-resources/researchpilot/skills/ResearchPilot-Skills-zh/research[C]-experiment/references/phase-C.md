# 阶段 C 详细流程：实验设计

## 阶段 C：实验设计

### 触发

阶段 B 用户确认 idea 后自动进入。

> **实验设计原则**：所有实验设计的第一目的是**严格证明 idea 的有效性**，不为迁就资源而削减实验。因此**不在设计前收集 GPU/训练时间等资源约束**，而是在实验方案完成后给出一份资源预估（见 C-6），供用户参考。资源不是设计的约束条件。

### C-1 Baseline 文献与代码精读

从 `idea_report.md` Part 2 Method 章节中提取所有选定的 baseline，向用户展示精读计划并请求确认：

```
在设计实验之前，我打算精读以下 baseline 的论文和代码仓库，
以了解它们如何设计实验，避免实验设计与领域惯例脱节：

| # | Baseline | 论文 | GitHub | 精读目的 |
|---|---------|------|--------|---------|
| 1 | {名称} | {标题} [n] | {repo 链接 或 暂未找到} | {一句话：为什么读这个——它代表了哪类方法，或实验设计有什么参考价值} |
| 2 | {名称} | ... | ... | ... |

确认后我将逐一读取，整理为 Part 3 第 0 节。
```

等待用户确认（可补充或删减列表）。

确认后，对每个 baseline 执行：
1. 下载论文 PDF（已在 `docs/papers/` 中的直接读取，否则按论文下载逻辑获取）
2. 读取 GitHub README 及核心训练脚本（若仓库可访问）
3. 从论文和代码中提取：
   - 使用了哪些数据集，如何划分（比例 / 官方划分 / 交叉验证）
   - 设计了哪些实验（主实验、消融、附加），每个实验的目的
   - 对比了哪些模型，模型选取逻辑
   - 使用了哪些评估指标，计算方式
   - 关键超参数设置（batch size、lr、训练轮数等）

提取完成后，向用户展示摘要确认无误，再进入 C-2。

### C-2 检索领域实验惯例

搜索同领域近 3 年论文，结合 C-1 的精读结果，归纳：
- 各 baseline 共同使用的数据集（视为该领域标准 benchmark）
- 各 baseline 共同使用的评估指标（视为领域标准指标）
- 消融实验的典型设计模式（哪些组件通常需要消融）
- 结果表格规范（是否报告标准差、是否多次运行取均值）

### C-3 数据与代码可获取性核实

> 只核实"实验能否做得成"，即数据和 baseline 代码能否拿到；**不核实 GPU 显存等资源是否够用**（资源在 C-6 仅作事后预估，不构成约束）。

检查两项（任一不通过，暂停并告知用户）：
- **数据集**：下载链接可访问，权限公开（需申请的注明申请方式）
- **Baseline 代码**：仓库可访问，框架兼容，无代码则标注"需自行实现"

可获取性摘要写在 Part 3 第 0 节末尾：
```markdown
### 数据与代码可获取性摘要
| 项目 | 状态 | 备注 |
|------|------|------|
| 数据集 {名称} | ✅/⚠️/❌ | {说明} |
| Baseline {名称} 代码 | ✅/⚠️/❌ | {说明} |
```

### C-4 提出实验大纲，与用户确认

> 在动笔写 Part 3 完整内容之前，**先给出一份实验大纲，与用户交互确认**。大纲是骨架，确认后再展开成正文，避免在错误的实验框架上写一大堆细节。

向用户展示大纲，包含以下内容：

```
在正式撰写实验设计前，先和你确认实验大纲：

**数据集**：计划用哪些数据集（主实验 / 附加实验分别用什么），为什么选这些。

**实验列表与设计逻辑**：
| # | 实验 | 类型 | 设计意图（为什么这样设计、支撑核心方法的哪一点）| 参与模型数 | 核心 baseline | 选它作核心的理由 |
|---|------|------|----------------------------------|----------|--------------|---------------|
| 1 | 主实验 | 主实验 | …… | {N} | {模型} | {为什么以它为核心} |
| 2 | 消融：w/o {模块} | 消融 | …… | {N} | — | — |
| 3 | {领域标配附加实验} | 附加·必做 | …… | {N} | {模型} | …… |

**可选的扩展实验**（补足工作量，供你挑选）：
- 选项 1：{实验名}——验证 {属性}，{一句话设计意图}
- 选项 2：{实验名}——验证 {属性}，……
- 选项 3：……
（从多个角度尽量多列，你可以勾选要做哪些）

确认大纲或告诉我需要调整的地方；选定扩展实验后，我再展开撰写完整的 Part 3。
```

与用户来回交互，直到大纲（含选定的扩展实验）确认。

### C-5 生成 idea_report.md Part 3

按 `references/document-formats.md` 中的 Part 3 模板，依据已确认的大纲生成完整内容，追加到 `idea_report.md` 末尾。

- 第 0 节（Baseline 实验调研）直接由 C-1 精读结果填入
- 第 1 节起的实验设计基于第 0 节归纳的领域惯例、按 C-4 确认的大纲展开
- **每个实验必须填写"为什么这样设计"字段**，说明该实验对支撑核心方法的意义
- **参与评测的模型表必须逐个说明**：与本文方法的区别、纳入意义、出处论文（仅展示，不参与选择决策），并指明核心 baseline 及其理由
- 附加实验分"领域标配（必做）"和"扩展实验（用户已选）"两类
- 每个实验设计决策用 `>` 说明理由，有论文依据的附引用编号

### C-6 实验设计完成后给出资源预估

实验方案全部设计完成后，基于模型规模、数据量、实验组数，给出一份**资源预估**，作为 Part 3 末尾的参考信息（**仅供参考，不是设计约束**）：
```markdown
### 资源预估（参考）
| 实验 | 预估显存 | 预估单次训练时长 | 实验组数 |
|------|---------|---------------|---------|
| 主实验 | ~{N}GB | ~{N} 小时/数据集 | {N} |
| 消融实验 | ~{N}GB | ~{N} 小时/变体 | {N} |

> 以上为粗略估算，实际资源消耗取决于具体实现与硬件。若资源受限，可在不影响有效性证明的前提下，优先跑核心实验、分批运行。
```

### C-7 每次输出后询问确认

```
实验设计已生成（含资源预估，仅供参考）。你觉得现在的实验方案够完善了吗？
如果可以，我们进入实现设计阶段，生成详细的编码指南。
或者你有什么需要调整的地方？
```

### C-8 迭代优化（循环）

用户提出修改后，更新 Part 3 内容，重新询问确认。
用户确认后，进入阶段 D（见 `references/phase-implementation.md`）。
