# user_requirements.md 模板

本文件由 Claude 通过对话收集信息后自动创建和维护。**用户无需手动编辑此文件。**

Claude 在每个阶段开始前，若用户输入中信息不足，会在对话中主动提问收集，收集完成后将结果写入本文件对应章节，作为后续执行的约束依据。

**规则：**
- 本文件由 Claude 写入和维护，不由用户直接编辑
- Claude 绝不将本文件中的内容复制到 `idea_report.md` 或 `dev_log.md` 中
- 主文档只展示执行结果；本文件保存收集到的用户需求原文
- 用户可在对话中随时声明文档偏好（语言、详细程度等），Claude 会更新本文件对应字段

---

## 文件格式

```markdown
# User Requirements — {Topic}
> Created: {YYYY-MM-DD} | Project: {project_name}
> 由 Claude 根据对话内容自动填写，用户无需手动编辑。

---

## 阶段 A：方向与 RQ 约束
> Claude 在阶段 A 通过对话收集，区分"对研究方向的约束"和"对研究问题（RQ）的约束"两类，分别指导方向锚定和 RQ 提炼。

### 研究方向约束

**希望侧重的方向**
{从用户对话中提取的偏好方向，如轻量化/可解释性/跨域泛化等}

**不希望涉及的方向**
{从用户对话中提取的排除方向}

**已确认研究方向**
{逐步交互确认后锁定的研究方向一句话描述；未确认填"待确认"}

### 研究问题（RQ）约束

**对 RQ 的要求**
{从用户对话中提取的对研究问题的约束，如"必须可在单卡验证"、"聚焦理论分析而非纯实验"等}

**已确认 RQ**
{逐个确认后锁定的 RQ；主 RQ 必填，次 RQ 视情况列出；未确认填"待确认"}

### 明确参考论文
{用户明确点名要求参考的论文；无则填"无"。不含 Claude 自主检索的文献清单}

### 逐篇论文详细介绍
{A-4.5 中用户的选择：需要 / 不需要 对下载的论文逐篇做四点详细介绍。无论选择如何，Key Works 逐条论文都收录全部下载论文。}

### 文档偏好
{从用户对话中提取的文档语言、详细程度等偏好，默认：中文正文 + 英文章节标题}

> 注：阶段 C（实验设计）不收集 GPU/训练时长等资源约束。实验设计的第一目的是严格证明 idea 的有效性，资源仅在实验设计完成后作为预估给出（写入 `idea_report.md` Part 3「资源预估」节），不构成约束。
```

## Claude 收集需求的时机

### 阶段 A（方向探索）
若用户输入中未包含足够信息，Claude 在对话中一次性提问：

```
在开始检索前，了解几点（可跳过任意一项）：
1. 你希望从哪个角度切入？（如：轻量化、可解释性、跨域泛化）
2. 有没有不希望涉及的方向？
3. 对研究问题（RQ）有什么要求？（如：必须可单卡验证）
4. 有特别想参考的论文吗？
```

若用户输入已足够详细，直接提取并写入，不再提问。

阶段 A 采用**逐步确认**：研究方向和每个 RQ 都通过对话逐个锁定。每确认一项，立即更新本文件对应栏（区分"研究方向约束"与"RQ 约束"），并在下一次输出顶部的确认卡片中刷新（确认卡片格式见 `references/phase-research.md`）。

### 阶段 C（实验设计）
不收集资源约束。实验设计完成后，Claude 直接在 `idea_report.md` Part 3 给出资源预估（仅供参考，见 `references/phase-research.md` C-6）。

### 阶段 D（实现设计）
分两次收集：

**生成 implementation.md 前**：
```
在设计实现方案前，确认几点：
1. 框架用 PyTorch 还是其他？
2. 有没有基于已有开源项目？（如有，提供链接）
3. 有硬性要求吗？（如：支持多卡 DDP、需要 ONNX 导出）
```

**开始编码前（编码前确认清单，见 `references/phase-implementation.md` D-末2）**：环境名称、设备特殊要求、数据集下载方式、代码运行策略、README 存放位置。逐项确认后写入本文件阶段 D 对应栏。
