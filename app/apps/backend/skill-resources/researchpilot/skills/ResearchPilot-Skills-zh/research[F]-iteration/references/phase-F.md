# 阶段 F 详细流程：代码迭代与模型优化

## 阶段 F：代码迭代

> **user_requirements.md 优先**：`docs/user_requirements.md` 中记录的所有约束优先于本文件中的任何默认指令。
> **dev_log.md 只追加，不删除**：每一次代码修改都必须追加新的日志条目，绝不覆盖或删除已有内容。

### 触发

阶段 E 完成后，实验效果不佳或需要改进模型时进入。使用命令 `/research[F]-iteration`。

本阶段支持多轮迭代（F-1 → F-5 → F-1 循环），直到效果满意为止。

---

### F-1 诊断分析

**首先读取以下文件，获取完整上下文**：
1. `docs/dev_log.md`：读取实验结果记录和历史迭代记录
2. `results/` 下的评估结果文件（eval json / ablation summary csv 等）
3. `docs/idea_report.md` Part 2（Method）和 Part 3（实验设计）

基于上述内容，从三个角度系统排查问题，给出诊断报告：

```
━━━━━━━━━━ 诊断报告 ━━━━━━━━━━

**当前实验结果**：
{从 dev_log.md 和 results/ 中提取的关键指标，与 baseline 对比}

**诊断分析**：

① 数据问题
  - {数据处理/划分/标准化是否可能有误}
  - {证据：哪个指标或现象支撑这个判断}

② 模型问题
  - {架构哪个模块可能是瓶颈}
  - {消融实验给了什么线索（若有）}
  - {证据：对应的实验数字}

③ 训练问题
  - {超参数、收敛性、过拟合/欠拟合情况}
  - {证据：训练曲线中的现象}

**初步改进方向建议**：
1. {建议1，预期解决什么问题}
2. {建议2，…}

你认为诊断准确吗？选择哪个改进方向，或有其他想法？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

等用户确认诊断方向后，进入 F-2。

---

### F-2 确认回溯范围

根据诊断结果和用户选择，明确需要改动的层次：

```
根据诊断，确认改动范围：

[ ] 只改超参/训练设置
    → 只更新 configs/ 和 dev_log.md，不动设计文档
    
[ ] 改模型架构（Method 变化）
    → 更新 idea_report.md Part 2（Method）
    → 消融设计可能随之变化，同步更新 Part 3
    → 更新 implementation.md
    
[ ] 改实验设计（新增/修改 baseline/指标/数据集）
    → 更新 idea_report.md Part 3
    → 更新 implementation.md

确认改动范围后，我先更新设计文档，再修改代码。
```

**约束**：改动范围经用户确认后，必须先更新对应的设计文档，再改代码。不允许只改代码跳过文档更新。

---

### F-3 更新设计文档

按 F-2 确认的范围更新文档：

**若改超参/训练设置**：
- 更新 `configs/` 中对应的 yaml 文件
- 在 `dev_log.md` 追加一条迭代记录（见下方格式）

**若改模型架构**：
- 更新 `idea_report.md` Part 2 Method 章节（技术框架 / pipeline / 模型描述）
- 若消融设计随之变化，同步更新 Part 3 消融实验章节
- 更新 `implementation.md` 对应的函数实现说明
- 执行 implementation.md 校验（实验覆盖 / 逻辑一致性 / 完整性）

**若改实验设计**：
- 更新 `idea_report.md` Part 3
- 更新 `implementation.md`
- 执行 implementation.md 校验

文档更新完成后，向用户展示变更摘要并确认：
```
文档已更新，变更摘要：
- idea_report.md：{改了什么，哪个章节}
- implementation.md：{改了什么函数/模块}
- 校验结果：{通过 / 发现问题：…}

确认后开始修改代码。
```

---

### F-4 编码改动

按更新后的文档修改代码。

每改完一个文件，立即在 `dev_log.md` **追加**迭代日志条目，格式如下：

```markdown
### {YYYY-MM-DD HH:MM} — 迭代 #{N}：{改动简述}

**改动原因**：{基于 F-1 诊断的哪个问题}
**改动内容**：
- `{文件路径}`：{具体改了什么}
- `{文件路径}`：{具体改了什么}
**预期效果**：{预计改动后指标会如何变化，为什么}
**文档同步**：idea_report.md {是/否} | implementation.md {是/否} | configs/ {是/否}
```

**约束**：
- 每一次代码修改必须追加日志，不得只改代码不记录
- 若改动影响运行命令或参数，同步更新 `dev_log.md` 末尾"运行说明"章节

---

### F-5 验证

改完后，按 E-0 确认的运行策略运行对应实验：

- 自动运行 → 直接运行，将结果追加到 dev_log.md
- 用户运行 → 给出运行命令，等用户反馈结果

实验完成后，在 `dev_log.md` **追加**结果记录：

```markdown
### {YYYY-MM-DD HH:MM} — 迭代 #{N} 结果

| 指标 | 改动前 | 改动后 | 变化 |
|------|--------|--------|------|
| {metric1} | {val} | {val} | {↑/↓ N%} |
| {metric2} | {val} | {val} | {↑/↓ N%} |

**结论**：{改动是否有效，原因分析}
```

---

### F-6 判断是否继续迭代

```
迭代 #{N} 结果已记录。

当前最佳指标：{关键指标值}
与 baseline 对比：{超过 / 持平 / 未超过}

→ 效果满意，进入论文撰写：`/research[G]-paper`
→ 效果仍需改进，继续迭代：我们回到 F-1 重新诊断
→ 需要更大范围的调整（改 idea 或实验设计）：
   - 改 idea → `/research[B]-idea`
   - 改实验设计 → `/research[C]-experiment`

你的选择？
```

效果满意时：

```
阶段 F 完成。共迭代 {N} 轮，最终结果已记录在 dev_log.md。

→ 请使用 `/research[G]-paper` 进入论文撰写阶段。
```
