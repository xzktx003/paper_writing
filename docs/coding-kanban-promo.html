AI Coding 时代的浏览器工作台

Coding Kanban：把你的多终端、多主机、多会话，收进同一张看板

相信大家在最近保活的 AI coding 时代，或多或少都会遇到这些问题： 会话越来越多、窗口越开越乱、远程文件越改越麻烦、机器一换环境就得重新连一遍。 Coding Kanban 给出的答案很直接——把本地终端、SSH 远端、tmux、文件浏览器、VS Code Web， 全部收口到一个可视化、可切换、可接管的浏览器工作台里。

1

你是不是还在为同时进行好几个 coding 会话而忙得手忙脚乱？

Copilot、Codex、Claude、shell 来回切，多个终端窗口反复跳，想接着输一句话都要先找对那个窗口。

2

你是不是还在为了服务器端文件浏览麻烦，而一直使用终端 ls、cd？

看目录、改配置、下日志、传文件，全靠命令行硬切，明明只是想“顺手看一眼、改一行”。

3

你是不是还在因为同时开太多 VS Code 和终端而让电脑内存报警？

一个项目一个窗口，一台服务器一个编辑器，再开几个终端标签，笔记本风扇直接起飞。

4

你是不是还在因为电脑一带回家，就得重连 VS Code、MobaXterm、terminal？

工作流碎在不同客户端里，换台机器、换个网络、换个位置，连接关系就得重新拼一次。

那么今天，我们给你一个更适合 AI Coding 的解决方案——Coding Kanban

它不是又一个“会开 shell 的页面”，而是一个面向 CLI Coding Agent 的多终端编排工作台： 你可以同屏观察多个会话、双击进入聚焦终端、直接打开远端文件浏览器、 在同一个页面里拉起或复用 VS Code Web，还能把现有 tmux 会话和历史工作目录重新接管进来。

一句话总结： 把“观察多个 Agent → 切到某个会话 → 继续输入 → 看文件 → 改文件 → 打开编辑器” 这一整串动作，压缩成一个连续工作流。

它到底解决了什么？

一个页面，统一管多个会话

本地 PTY、SSH 远端 PTY、tmux 会话、扫描到的历史目录、观察中的 VS Code 窗口，全部在一张宫格看板里统一展示。

一个工作流，覆盖终端 + 文件 + 编辑器

不是看板只负责“看”，编辑器只负责“改”。聚焦态里可以直接开文件浏览器和 VS Code Web， 终端上下文不用断。

一个浏览器，跨机器继续工作

更适合本地自托管、内网开发机、团队共享工作台这些场景。换电脑、换位置， 你的工作流不必再拆散重连。

1. 多 Agent 看板，不再靠窗口记忆工作

Coding Kanban 把会话从“标签页”升级成“看板卡片”。 每张卡片都能显示名称、主机、工作目录、状态、Agent 类型和终端缩略图； 你可以筛选、隐藏、恢复、重命名、关闭、终止 tmux，也能双击直接进入聚焦态。

换句话说，它不是让你“拥有更多终端”，而是让你终于能管理这些终端。

本地 PTY SSH 远端 tmux 接管 隐藏/恢复 聚焦工作流

2. 远端文件浏览，不用再把 ls / cd 当文件管理器

聚焦态下，内置文件浏览器可以直接打开本地文件系统，也支持通过 SSH/SFTP 访问远端开发机。目录树、面包屑、隐藏文件、筛选、排序、拖拽上传、下载、重命名、删除、chmod 一应俱全。

文本文件还能直接预览和编辑保存。对于“看日志、改配置、顺手修一行”的高频动作来说， 这比反复切回另一个终端或另一个 SFTP 工具高效得多。

本地文件 SSH / SFTP 文本预览 编辑保存 上传下载

3. VS Code Web 内嵌，不再每台机器都单独开一个编辑器

本地终端会话和 SSH 远端会话都能直接打开内嵌 VS Code Web。 对本地环境，它会优先复用 code-server，其次兼容 openvscode-server；对 SSH 远端，它会通过 SSH 启动或复用远端 code-server，再统一代理到当前后端的 /vscode/。

这意味着：你不必为了每台开发机都单独开一个完整的桌面 VS Code 窗口， 也不必在本机和远端编辑器之间来回跳。

本地 VS Code Web SSH 远端 VS Code Web 扩展目录复用 统一代理

4. tmux 扫描、接管与恢复，让“已有会话”重新变得可管理

很多团队的问题不是“不会创建新会话”，而是“已有 tmux 和历史工作目录越来越多，却没人能管理”。 Coding Kanban 不只支持新建，还支持扫描本地和 SSH 远端 tmux，把已存在的 pane 接进来看板继续控制。

同时，它还能扫描工作目录、识别 Copilot 的 session-state，并尝试把扫描结果和 tmux pane 合并，减少重复卡片。这让“重新捡起昨天的工作”比“从头重建环境”更容易。

适合谁用？

本地自托管开发者

希望把自己的 CLI Agent、tmux、远端开发机、文件浏览和编辑器统一在一个浏览器工作台里。

内网开发机 / GPU 服务器用户

习惯在固定开发机上跑长任务，但不想每台机器都开完整的桌面远程编辑环境。

多 Agent 并行的 AI Coding 团队

需要同时观察多个会话状态、快速切换输入、接管已有工作目录和远端 tmux 的团队工作流。

安装方法：一句话交给你的大模型

如果你现在就在用大模型协助开发，最省事的方式其实非常简单：直接让它把仓库依赖装好并启动。 注意，仓库里实际使用的脚本名是 ./scripts/restart-dev.sh。

你可以直接这样说

把 http://10.10.1.58/xing.hu/coding_kanban 这个仓库的依赖安装好，
然后运行 ./scripts/restart-dev.sh

脚本会自动清理占用端口、启动后端、启动前端，并打印出实际 Local / Network 地址，方便你在局域网里直接打开。

它会帮你准备什么

  • Node.js + pnpm 依赖安装
  • 前后端开发服务同时启动
  • 局域网可访问的前端地址
  • HTTP 浏览器工作台
  • 后续可接入 SSH 主机、tmux、文件浏览器和 VS Code Web
结语

如果你已经进入“一个人同时保活多个 AI coding 会话”的工作状态， 那么你真正缺的往往不是更多终端，而是一个能把这些终端、文件和编辑器组织起来的工作台。

Coding Kanban 做的正是这件事：让本地与远端、终端与编辑器、已有会话与新会话，全部回到同一个浏览器界面里。 少一点窗口切换，多一点真正连续的编码流。
